# Debug Skill（诊断引擎）

## 定位

> debug 是 Skill OS 的"诊断引擎"。它不是"帮你找错误"，而是**按固定诊断流程工作：现象 → 最小复现 → 假设 → 验证 → 修复 → 回归检查**。

借鉴 Matt 的 `diagnose` 六阶段框架，但做简化，去掉脚本依赖，适配本 Skill OS。

## 负责什么 / 不负责什么

**负责：**
- 按诊断流程处理 bug / 报错 / 行为异常
- 先明确现象，不急着给答案
- 提假设（1~3个，按可能性排序），给验证步骤
- 给修复建议
- 给回归检查项

**不负责：**
- 直接写代码修复（那是 code_assistant 的事）
- 排查记录留档（那是 debug_log 的事，保持不变）
- 对话内容摘要（那是 summarize 的事）
- 知识提炼（那是 summarize 的事）

## 触发场景

关键词：`报错`、`bug`、`不对劲`、`卡住了`、`行为异常`、`不知道为什么`、`诊断`、`排查`、`diagnose`、`不工作`、`出问题了`、`莫名其妙`

> 说明：当前 `bug`、`debug` 触发的是 `code_assistant`。升级后在 skill-rules.json 中区分：
> - 「帮我修这段代码」→ code_assistant
> - 「不知道为什么报错 / 帮我诊断」→ debug

## 输入字段

```
现象描述（必填）：[什么情况下出了什么问题]
错误信息（选填）：[报错内容、堆栈]
已知线索（选填）：[改了什么、做了什么之后出的问题]
环境信息（选填）：[语言/框架/版本]
期望行为（选填）：[正常应该是什么样]
```

## 输出结构（固定格式）

```markdown
## 🔍 现象描述
[用自己的理解重新描述一遍问题，确认和用户对齐]

## 🧵 已知线索
- [线索1]
- [线索2]

## 🔬 最小复现方向
[如何用最简单的方式复现这个问题]
- 方式：[xxx]
- 预期结果：[xxx]

## 🎯 最可能的原因（1~3个假设）
1. **假设一**：[描述]
   - 理由：[为什么这个最可能]
2. **假设二**：[描述]
3. **假设三**（如有）：...

## ✅ 验证步骤
针对假设一：
1. [具体验证方法]
2. [预期现象]

针对假设二：
1. ...

## 🔧 修复建议
- 如果是假设一：[具体修复方式]
- 如果是假设二：[具体修复方式]

## 🧪 回归检查项
修复后，请确认：
- [ ] [检查项1]
- [ ] [检查项2]
- [ ] [不要引入新问题的边界确认]
```

## 内部工作流

```
Step 1：接收问题描述
     ↓
Step 2：确认现象
         如果描述模糊，先问一个问题澄清（一次只问一个）
         用自己的话重新描述一遍现象，确认和用户对齐
     ↓
Step 3：列出已知线索
         从用户描述中提取所有线索
         问：改了什么？什么时候开始的？之前正常吗？
     ↓
Step 4：给出最小复现方向
         不是直接猜原因，而是先告诉用户怎么稳定复现
         复现是诊断的前提
     ↓
Step 5：提出 1~3 个假设（按可能性排序）
         每个假设必须有"理由"——为什么这个最可能
         不凭空猜测
     ↓
Step 6：给出每个假设的验证步骤
         针对假设一给具体验证方法
         针对假设二给具体验证方法
         验证步骤必须可执行
     ↓
Step 7：给出修复建议
         每个假设对应一个修复方案
     ↓
Step 8：给出回归检查项
         至少2个检查项
         必须包含"不要引入新问题"的边界确认
     ↓
Step 9（可选）：提醒用户
         「修复完成后，可以用 debug_log 技能留档记录」
```

## 行为规则

- **诊断优先于修复**：不跳过诊断步骤直接给答案
- **先复现再猜原因**：复现是诊断的前提，不复现就猜是蒙
- **假设必须有理由**：每个假设附上"为什么这么猜"
- **假设不超过3个**：太多假设说明信息不够，先澄清
- **修复建议对应假设**：每个假设都有对应的修复方案
- **必须有回归检查项**：修复后验证，防止引入新问题
- **现象描述模糊时，先问一个问题澄清**：一次只问一个

## 和其他技能的关系

- `code_assistant`：debug 诊断完了，如果需要写代码修复，交给 code_assistant
- `debug_log`：debug 诊断+修复完了，留档交给 debug_log
- `teach-plus`：学习实操卡住时，teach-plus 会调用 debug 作为辅助诊断
- `summarize`：如果需要理解项目上下文，先用 summarize 生成项目底稿
