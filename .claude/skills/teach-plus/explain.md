# Teach-Plus / Explain 模式

## 职责

把学习对象讲清楚，帮用户建立"这是什么、为什么重要、先看什么"的理解框架。

## 适用场景

- "我想学这个仓库 / 项目 / 技能"
- "把这个文档讲给小白听"
- "帮我梳理这个知识体系"
- "这个系统是怎么工作的？给我讲一下主线"
- 用户第一次接触某个学习对象，需要建立整体理解框架

## 输入

### 必需输入

1. **学习对象**：仓库 / 文档 / 技能 / 概念（至少一个）
2. **用户当前目标**：入门 / 看懂 / 能改 / 能复述 / 能讲给别人听（未指定时默认"会用"）

### 可选输入（优先消费）

3. **summarize 产出的学习底稿**（briefing）：如果已有，直接复用其中的结构化信息
4. **用户当前水平**：零基础 / 看过一点 / 会一点（未指定时默认"看过一点"）

## 输出结构

按以下结构输出理解框架：

```markdown
## 📖 学习对象理解框架

### 1. 一句话定义
[用一句话说清楚这个学习对象是什么、解决什么问题]

### 2. 核心概念 / 核心模块
- 概念A：[是什么，在整个体系中的位置]
- 概念B：[是什么，在整个体系中的位置]
- ...

### 3. 主线结构 / 运行链路
[从入口到出口的主流程，用文字或简单图示描述]
[如果是仓库：从目录结构到核心文件到调用关系]
[如果是技能系统：从输入到 intent 判定到 workflow 到输出]

### 4. 必须先懂的前置点
[学这个之前，需要先知道什么？列出 2-4 个前置知识/概念]
[如果用户缺少前置，建议先补什么]

### 5. 最容易卡住的点
- 卡点1：[是什么，为什么容易卡]
- 卡点2：[是什么，为什么容易卡]
- ...

### 6. 推荐学习顺序
1. [第一步：看什么 / 读什么]
2. [第二步：理解什么]
3. [第三步：动手做什么]
4. ...

### 7. 下一步建议
- 如果理解了上述内容 → 进入 **practice 模式**，生成每日学习单
- 如果还有概念不清 → 回到 **summarize** 补充学习底稿
- 如果前置知识不足 → 先补前置，再回来
```

## 与其他技能的关系

| 技能 | 在 explain 中的角色 |
|------|-------------------|
| **summarize** | explain 优先消费 summarize 产出的 briefing（学习底稿），从中提取核心概念和主线 |
| **planning** | explain 不负责制定阶段计划；如果用户需要长期学习路线，引导到 planning |
| **practice** | explain 的"下一步建议"中引导用户进入 practice 模式 |

## 行为规则

- **先找底稿**：检查是否已有 summarize 产出的学习底稿；如有，直接复用
- **无底稿时**：如果学习对象是仓库/文档，先自行做结构化分析（等效于调用 summarize）
- **控制深度**：explain 的目标是"建立框架"，不是"穷尽细节"；细节留给 practice
- **输出后引导**：每次 explain 输出后，明确告诉用户下一步是进入 practice 还是补充 briefing
- **语言适配**：根据用户水平调整解释深度；零基础多用类比，有基础多用结构图
- **知识沉淀**（v5 强制）：explain 输出后，将"学习对象理解框架"交给 `knowledge-asset`（knowledge-note 模式）沉淀到 `knowledge/knowledge-notes/`
