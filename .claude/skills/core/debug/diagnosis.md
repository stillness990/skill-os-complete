# Debug: Diagnosis（诊断流程）

## 定位

diagnosis 是 debug 技能的核心诊断流程文件。它定义了从"看到问题"到"给出修复建议"的完整诊断步骤。

## 诊断流程（8 步）

```
Step 1：接收问题描述
    ↓
Step 2：确认现象
    - 如果描述模糊，先问一个问题澄清（一次只问一个）
    - 用自己的话重新描述一遍现象，确认和用户对齐
    ↓
Step 3：列出已知线索
    - 从用户描述中提取所有线索
    - 追问：改了什么？什么时候开始的？之前正常吗？
    ↓
Step 4：给出最小复现方向
    - 不是直接猜原因，而是先告诉用户怎么稳定复现
    - 复现是诊断的前提
    ↓
Step 5：提出 1~3 个假设（按可能性排序）
    - 每个假设必须有"理由"——为什么这个最可能
    - 不凭空猜测
    ↓
Step 6：给出每个假设的验证步骤
    - 针对每个假设给具体验证方法
    - 验证步骤必须可执行
    ↓
Step 7：给出修复建议
    - 每个假设对应一个修复方案
    - 标注哪些需要代码修改，交给 code_assistant
    ↓
Step 8：给出回归检查项
    - 至少 2 个检查项
    - 必须包含"不要引入新问题"的边界确认
```

## 关键原则

1. **诊断优先于修复** — 不跳过诊断步骤直接给答案
2. **先复现再猜原因** — 复现是诊断的前提，不复现就猜是蒙
3. **假设必须有理由** — 每个假设附上"为什么这么猜"
4. **假设不超过 3 个** — 太多假设说明信息不够，先澄清
5. **现象描述模糊时，先问一个问题澄清** — 一次只问一个
6. **修复建议对应假设** — 每个假设都有对应的修复方案
7. **必须有回归检查项** — 修复后验证，防止引入新问题

## 常见诊断模式

### 报错诊断
1. 读错误信息 → 定位报错行
2. 检查该行上下文 → 看变量/函数/依赖
3. 假设：语法错 / 类型错 / 依赖缺失 / 边界条件

### 行为异常诊断
1. 明确"预期行为" vs "实际行为"
2. 回溯最近的改动
3. 假设：逻辑变更 / 配置变化 / 环境变化

### 性能问题诊断
1. 确认性能瓶颈的具体表现（慢在哪里）
2. 检查是否存在 N+1 查询 / 大循环 / 阻塞 IO
3. 假设：算法复杂度 / IO 阻塞 / 资源竞争

## 输出

诊断结果按 `debug-protocol.md` 格式输出（见 `.claude/protocols/debug-protocol.md`）。
