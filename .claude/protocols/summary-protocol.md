# Summary Protocol（项目底稿协议）

`summarize/basic` 模式的标准输出协议。供 `summarize` 技能遵循，也供 `planning`/`debug` 等下游消费者理解输入结构。

---

## 协议字段

### 核心主题
- 一句话说清这个项目/模块/文档是什么
- 不超过 50 字

### 关键模块 / 关键事实
- 列出 3~8 个核心模块或关键事实
- 每个条目一行，附一句话描述
- 按重要性排序

### 重要细节
- 3~5 个不可忽略的细节
- 包括：关键约束、特殊行为、隐藏依赖

### 风险 / 缺失点
- 至少 1 个当前存在的问题或风险
- 如果没有明显风险，写"未发现明显风险"
- 标注已知的缺失信息

### 下一步建议
- 2~3 个可执行的下一步建议
- 按优先级排列
- 每个建议指向一个具体技能或行动

---

## 输出模板

```markdown
## 📦 核心主题
[一句话描述]

## 🧩 关键模块 / 关键事实
- [模块/事实 1]：[一句话描述]
- [模块/事实 2]：[一句话描述]
- [模块/事实 3]：[一句话描述]
...

## 📌 重要细节
1. [细节]
2. [细节]
3. [细节]

## ⚠️ 风险 / 缺失点
- [风险/缺失 1]
- [风险/缺失 2]

## 👉 下一步建议
1. [建议 1] → [指向的技能/行动]
2. [建议 2] → [指向的技能/行动]
```

---

## 使用场景

| 场景 | 示例 |
|------|------|
| 仓库概览 | "总结一下这个仓库" |
| 文档整理 | "帮我把这篇文档提炼要点" |
| 对话总结 | "总结一下我们刚才讨论的内容" |
| 日志概览 | "把今天的日志梳理一下" |

## 与其他协议的关系

- `briefing-protocol.md` — basic 的"深度版本"，供 planning/debug 消费
- `plan-protocol.md` — planning 消费 summary/briefing 后产出
- `debug-protocol.md` — debug 消费 summary/briefing 后产出
