# Plan Protocol（计划协议）

`planning` 技能的标准输出协议。供 `planning` 遵循，也供 `task_ledger` 和 `teach-plus` 理解输入结构。

---

## 协议字段

### 目标
- 一句话说清要完成什么
- 必须可验证（"完成XX功能"而非"了解XX"）

### 当前阶段判断
- 标注当前处于哪个阶段
- 如果尚未开始，标注"Phase 0（准备）"

### 阶段拆解
- 2~5 个阶段
- 每个阶段必须包含：名称、时间范围、目标、产出物、关键任务（1~3项）
- 阶段之间必须有明确的依赖或顺序关系

### 今日最小行动
- 1~3 个具体动作
- 每个 ≤ 30 分钟
- 必须可执行（"打开XX文件，修改YY"而非"实现ZZ功能"）
- 状态差也能做

### 依赖 / 阻塞项
- 标注当前被什么阻塞
- 标注需要先完成什么

### 下一步建议
- 明确下一步做什么
- 指向具体技能或操作

---

## 输出模板

```markdown
## 🎯 目标
[一句话，可验证]

## 📊 当前阶段
[Phase X：名称]（状态：进行中 / 未开始）

## 🗂 阶段计划
### 阶段一：[名称]（第 x~y 天 / 第 x 周）
- **目标**：[xxx]
- **产出物**：[xxx]
- **关键任务**：
  1. [xxx]
  2. [xxx]

### 阶段二：[名称]
...

## 📅 今日最小行动
1. [具体动作，≤30分钟]
2. [可选]
3. [可选]

## 🚧 依赖 / 阻塞项
| 阻塞项 | 类型 | 解决方案 |
|--------|------|---------|
| [xxx] | 外部依赖 / 信息缺失 / 技术阻塞 | [xxx] |

## 🧭 下一步建议
1. [建议 1] → [指向的技能]
2. [建议 2] → [指向的技能]
```

## 两种模式

### project 模式（项目交付）
- 用于：仓库重构、功能开发、项目升级、任务拆解
- 输出侧重：阶段计划 + 产出物 + 阻塞项
- 必含：依赖关系 + 阻塞项表格

### learning 模式（学习计划）
- 用于：学习仓库、学习技能体系、学习技术主题
- 输出侧重：学习主线 + 卡点预估 + 最小行动
- 必含：学习顺序理由 + "最容易卡的点"

## 与 task_ledger 的集成

planning 产出的每个阶段的关键任务，应转换为 `task_ledger` 中的 task 条目：

```json
{
  "task_id": "tsk_YYYYMMDD_xxx",
  "title": "[从 plan 的关键任务提取]",
  "workflow": "delivery_pipeline | learning_pipeline",
  "status": "queued",
  "source": "planning",
  "next_action": "[从 plan 的今日最小行动提取]"
}
```

planning 自身不负责长期状态管理，状态管理由 `task_ledger` 系统层承担。
