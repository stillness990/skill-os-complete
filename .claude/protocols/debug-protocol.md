# Debug Protocol（诊断协议）

`debug` 技能的标准输出协议。供 `debug` 遵循，也供 `code_assistant` 和 `debug_archive` 理解输入结构。

---

## 协议字段

### 问题现象
- 用自己的话重新描述问题，确认和用户对齐
- 包含：什么情况下、出了什么问题、预期行为是什么

### 影响范围
- 影响哪些功能/模块/用户
- 严重程度：低 / 中 / 高

### 最小复现条件
- 如何用最简单的方式复现
- 包含：复现步骤 + 预期结果

### 初步假设（1~3 个，按可能性排序）
- 每个假设附理由
- 不超过 3 个（太多说明信息不够）

### 验证步骤
- 针对每个假设的具体验证方法
- 每个步骤预期看到什么

### 根因判断
- 确认后的根本原因
- 一句话说清

### 修复建议
- 如果是假设一：[具体修复方案]
- 如果需要代码修改：列出需交给 `code_assistant` 的改动点

### 回归验证清单
- 至少 2 个检查项
- 必须包含"不要引入新问题"的边界确认

---

## 输出模板

```markdown
## 🔍 问题现象
[用自己的理解重新描述问题]

## 📊 影响范围
- 影响模块：[xxx]
- 严重程度：低 / 中 / 高

## 🔬 最小复现
- **步骤 1**：[xxx]
- **步骤 2**：[xxx]
- **预期结果**：[xxx]

## 🎯 初步假设
1. **假设一**：[描述]
   - 理由：[为什么这个最可能]
   - 可能性：高 / 中 / 低
2. **假设二**：[描述]
   - 理由：[xxx]
   - 可能性：高 / 中 / 低

## ✅ 验证步骤
### 针对假设一
1. [验证方法]
2. 预期：[xxx]

### 针对假设二
1. [验证方法]
2. 预期：[xxx]

## 🔧 根因判断
[一句话根因]

## 🛠 修复建议
- 如果是假设一：[具体修复]
- 如果需要代码修改 → 交给 `code_assistant`：
  - 文件：[xxx]
  - 改动点：[xxx]

## 🧪 回归验证清单
- [ ] [检查项 1]
- [ ] [检查项 2]
- [ ] 边界确认：修复没有引入新的异常行为
```

---

## 与其他协议/技能的关系

| 下游 | 关系 |
|------|------|
| `code_assistant` | debug 诊断完后，如需代码修改，将改动点交给 code_assistant |
| `debug_archive` | 诊断+修复完成后，留档到 `system/debug_archive/` |
| `summarize/briefing` | 复杂问题场景下，可先用 summarize/briefing 生成背景底稿 |
| `task_ledger` | debug 发现的问题可转成 task 条目 |
